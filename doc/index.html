A compact self-contained realtime audio-synthesis engine

Lyd is a signal processing language and engine for audio processing. Suitable for realtime effect and instrument synthesis.

Features

  • library/engine
    • Small binary footprint
    • language binding friendly small C API
    • audio synthesis language, audio-shaders or "GLSL for audio", lyd uses a language but the vm could just as easily support nodes.
    • Extendable virtual machine.
    • 32bit float processing
    • patch databank (preloaded with FM-synthesis biased GM set)
    • Loading of PCM data from wav (using callback handlers and libsndfile)
    • Audio output drivers (outside core) for: ALSA, and jack
    • MIDI sequencer - for playback of midi tracks (timing is broken)
  • audio daemon
    • MIDI/OSC control

API documentation

The public API is defined in lyd.h, examples of it is used can be seen in some examples. The lyd language is documented in the lyd language reference..

Lyd binary / audio daemon

In addition to installing a library and public headers, lyd installs a commandline audio synthesis daemon that can be operated via MIDI or OSC (depending on library availability at compile time.

The OSC support needs rethinking, and probably re-integration with the midi support. Allowing repatching instruments used with MIDI on the fly from OSC.

The lyd binary will try to play back midi files passed in the first argument.

How it works

The following is a technical summary of what lyd actually is, it is not neccesary to understand any of it to make use of lyd's language and API.

Lyd uses a Top Down Operator Precedence parser to and a small compiler, lyd-compiler.c, to generate op-codes for it's virtual machine.

The op-code process small chunks of data (up to 64 samples, this size needs further investigation and perhaps needs tuning according to the complexity of the program) per processing cycle. This chunking outperforms a minimalistic sample by sample implementation almost by double even without the SIMD optimizations this refactoring was done in preparation for.

The op-codes in compiled LydPrograms are in processing order and arguments to the opcodes are either be positive literals or negative integer back-pointers to the outputs of preceding opcodes.

When a compiled program is instantiated to an active LydVoice the opcodes gain a processing context containing local storage for generated output, pointers to input arguments and a pointer used for ops with local state. When running the op-codes are only reading and modifying their own local state.

The values for interpolated arguments are written into the arrays for literal values in the relevant op-codes.

The op-codes of lyds virtual machine, and the functions and operators of the lyd language are all defined in lyd-ops.inc Op-codes added following the pattern in this file are added both to the compiler, the vm and the language reference.
